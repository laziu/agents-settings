# Ideation Examples

Use as rhythm references, not scripts.

## Example 1: Product Idea — "Help small restaurants compete with delivery platforms"

Reframe:
> How might we help independent restaurants keep direct relationships with their best customers without paying marketplace commissions?

Sharpening questions:
1. Which pain matters most: acquisition, ordering, delivery, loyalty, margin?
2. Is the user owner, diner, or both?
3. Constraints: software only, budget, operations?

Variations: direct ordering toolkit; pickup-first, no delivery; regular-customer reorder loop; restaurant collective; SMS "your usual?" flow.

Converged: focus on repeat customers ordering direct; SMS-first pickup-only MVP; avoid delivery, marketplace discovery, POS integration, dashboards.

## Example 2: Existing Feature — "Add real-time collaboration to our editor"

Reframe:
> How might we let multiple users work in one document without chaos?

Codebase-aware questions: same paragraph or separate sections? 2-5 users or larger? Competitive checkbox or differentiator?

Variations: presence only; block-level locking; async suggestions/comments; branch-and-merge; CRDT co-editing.

Converged depends on context:
- Competitive checkbox: block-level locking or presence first
- Differentiator: async suggestions or branch-and-merge
- Avoid character-level CRDT unless required

## Example 3: Process — "Retrospectives are stale"

Reframe:
> How might we make retros produce visible change instead of recurring theater?

Questions: same topics/voices, or no follow-through? Team size and dynamics? What did the last useful retro do differently?

Variations: one action item max; next retro reviews previous action; async written retro; anonymous topic collection; one-question retro; experiment-based retro.

Converged: fix output first (one action item, owner, deadline, review next retro); add anonymous input if quiet voices missing; avoid new tools or elaborate facilitation until follow-through works.

## What Good Sessions Do
- Reframe the problem, not just the solution
- Ask diagnostic questions before variations
- Explain why each variation exists
- Offer an opinion with rationale
- Pressure-test value, feasibility, differentiation
- Produce a concrete one-pager
- Make the Not Doing list explicit
- Adapt to product, codebase, or process context
